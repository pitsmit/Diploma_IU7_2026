\section{Выбор языков программирования и среды разработки}

Для разработки серверной части программно-алгоритмического комплекса был выбран язык
программирования \texttt{C++}~\cite{Cpp}, что обусловлено следующими причинами:
\begin{enumerate}
    \item встроенные возможности объектно-ориентированного программирования, позволяющие
создавать легко расширяемые приложения;
    \item совместимость с языком \texttt{C}, возможность прямого вызова функций монтирования устройств;
    \item поддержка многопоточности, необходимой для параллельной обработки событий;
    \item стандартная библиотека предоставляет множество контейнеров, упрощающих разработку.
\end{enumerate}

Для разработки клиентской части был выбран язык \texttt{TypeScript}~\cite{TypeScript} и фреймворк \texttt{Vue.js}~\cite{VueJS}. 

Язык \texttt{TypeScript} обеспечивает статическую типизацию, что позволяет уменьшить количество ошибок на 
этапе разработки и повысить удобство сопровождения кода.

Выбор \texttt{Vue.js} обусловлен компонентным подходом к разработке пользовательского интерфейса, что упрощает 
повторное использование элементов приложения и повышает удобство сопровождения проекта. Дополнительно 
фреймворк предоставляет встроенные средства реактивного обновления интерфейса, позволяющие эффективно 
отображать изменения состояния приложения в режиме реального времени.

Для управления глобальным состоянием приложения была выбрана библиотека \texttt{Pinia}~\cite{Pinia}, являющаяся
официальным инструментом управления состоянием для \texttt{Vue.js}. Использование \texttt{Pinia} позволяет централизованно
хранить и обрабатывать данные приложения, упрощая взаимодействие между компонентами.

Для стилизации пользовательского интерфейса был выбран препроцессор \texttt{SASS}~\cite{SASS}, расширяющий возможности
стандартного \texttt{CSS (Cascading Style Sheets)}. Использование \texttt{SASS} позволяет применять переменные и повторно используемые стили, 
что упрощает разработку и сопровождение интерфейса приложения. Дополнительно препроцессор способствует повышению 
читаемости и структурированности стилей, а также уменьшает дублирование кода.

В качестве среды разработки был выбран \texttt{Visual Studio Code}~\cite{VSCode} --- кроссплатформенная среда,
функциональность которой может быть увеличена с использованием дополнительных расширений.

\section{Выбор средств тестирования кода}

Для тестирования серверной части приложения был выбран фреймворк \texttt{GoogleTests}~\cite{GoogleTests}. 
Данный продукт позволяет создавать \texttt{unit}-тесты, \texttt{e2e}-тесты и интеграционные тесты. Разработан специально 
для \texttt{C++}, и использует стандартный его синтаксис. Позволяет реализовывать \texttt{Mock}-объекты для изоляции 
зависимостей при тестировании отдельных компонентов системы~\cite{Mocks}. Также фреймворк поддерживает 
параметризованные тесты, что позволяет проверять поведение методов на различных наборах входных 
данных без дублирования кода тестов. Для выполнения системно-зависимых тестов, таких как монтирование,
тесты и приложение запускаются в контейнерах \texttt{Docker}~\cite{Docker}.

Для тестирования клиентской части приложения была выбрана библиотека \texttt{Vitest}~\cite{ViTest}. 
Данная библиотека предназначена для тестирования \texttt{JavaScript}- и \texttt{TypeScript}-приложений, а также 
является нативной для фреймворка \texttt{Vue.js}. \texttt{Vitest} поддерживает создание \texttt{unit}- и компонентных тестов, 
предоставляет встроенные средства мокирования функций и модулей. Дополнительно библиотека поддерживает 
параллельное выполнение тестов, что сокращает время тестирования клиентской части приложения.

\section{Обеспечение связи между клиентом и сервером}

Для связи клиента и сервера используется протокол \texttt{HTTP (HyperText Transfer Protocol)}~\cite{HTTP} и механизм 
\texttt{WebSocket}~\cite{WebSocket}.

Протокол \texttt{HTTP} является протоколом прикладного уровня модели \texttt{OSI(Open Systems Interconnection)} и 
предназначен для обмена данными между клиентом и сервером по схеме <<запрос-ответ>>. Клиент формирует 
\texttt{HTTP}-запрос, сервер обрабатывает его и возвращает ответ, содержащий необходимые данные. Протокол широко 
применяется при передаче гипертекстовых документов и поддерживается большинством современных программных и аппаратных 
средств. В разрабатываемой программе используется, например, для получения списка подключённых устройств на клиенте.

Протокол \texttt{WebSocket} используется для организации постоянного двустороннего соединения между клиентом и сервером 
в режиме реального времени. В отличие от \texttt{HTTP}, \texttt{WebSocket} обеспечивает асинхронный обмен сообщениями и уменьшает 
накладные расходы на передачу данных за счёт отсутствия необходимости повторного создания соединения для каждого 
запроса. Установление \texttt{WebSocket}-соединения начинается с \texttt{HTTP}-запроса, после чего взаимодействие продолжается 
по отдельному протоколу \texttt{WebSocket}. В разрабатываемой программе используется для передачи на клиент информации о подключении
и извлечении устройств.

\section{Выбор программных средств для работы с устройствами и файловыми системами}

Для взаимодействия с устройствами и файловыми системами в операционной системе \texttt{Linux} были выбраны
библиотеки \texttt{libudev}~\cite{libudev}, \texttt{libmount}~\cite{libmount} и \texttt{libblkid}~\cite{libblkid}. 
Использование данных библиотек позволяет применять стандартные механизмы \texttt{Linux} для работы с 
\texttt{USB}-устройствами, операциями монтирования и определением типов файловых систем в пользовательском пространстве.

Библиотека \texttt{libudev} используется для получения информации об устройствах и отслеживания событий
их подключения и отключения. 

Библиотека \texttt{libmount} применяется для чтения файла \texttt{/proc/self/mountinfo} и определения 
режимов монтирования и точек монтирования. Выбор \texttt{libmount} обусловлен тем, что библиотека 
предоставляет унифицированный интерфейс для работы с данными монтирования \texttt{Linux} и 
позволяет избежать ручного чтения системных файлов.

Библиотека \texttt{libblkid} применяется для получения типа файловой системы 
\texttt{USB}-накопителя перед выполнением операций монтирования.

\section{Структура проекта серверной части комплекса}

Исходный код приложения, а также вспомогательные и служебные элементы были логически 
разделены по функциональному признаку следующим образом:

\begin{itemize}[label=---]
    \item \texttt{api-stubs/api/} --- содержит в себе обработчики \texttt{HTTP} запросов;
    \item \texttt{external/} --- директория для сторонних библиотек. Содержит \texttt{ixwebsocket} 
    (библиотека для \texttt{WebSocket}-соединений) и \texttt{spdlog} (библиотека для журналирования);
    \item \texttt{sql/schema} --- хранит скрипты для создания таблиц базы данных;
    \item \texttt{src/} --- исходный код приложения:
    \begin{itemize}[label=---]
        \item \texttt{commands/} --- содержит классы команд:
        \begin{itemize}[label=---]
            \item \texttt{Command} --- базовый класс команды;
            \item \texttt{GetWhiteListDeviceCommand} --- команда получения списка доверенных устройств;
            \item \texttt{AddDeviceToWhiteListCommand} --- команда добавления устройства в список доверенных;
            \item \texttt{DeleteDeviceFromWhiteListCommand} --- команда удаления устройства из списка доверенных;
            \item \texttt{PatchValidToDeviceCommand} --- команда изменения даты истечения доверия устройства;
            \item \texttt{GetCurrentConnectedDevicesCommand} --- команда получения списка подключённых в данный момент устройств;
            \item \texttt{CommandContext} --- вспомогательный класс, содержащий необходимые сервисы для команд.
        \end{itemize}
        \item \texttt{core/} --- содержит классы, реализующие основную работу по монтированию устройств и обработки событий:
        \begin{itemize}[label=---]
            \item \texttt{DeviceControlService} --- класс управляет обработкой события;
            \item \texttt{EventLoop} --- класс, отправляющий события из очереди на обработку;
            \item \texttt{EventQueue} --- класс, управляющий состоянием очереди событий;
            \item \texttt{IDeviceResolver} --- интерфейс класса получения параметров устройств и монтирования;
            \item \texttt{IMountSystem} --- интерфейс класса операций монтирования/размонтирования;
            \item \texttt{LinuxMountSystem} --- класс операций монтирования/размонтирования;
            \item \texttt{MountRecoveryService} --- класс обновления записей о монтировании при запуске приложения;
            \item \texttt{MountService} --- класс, управляющий операциями монтирования;
            \item \texttt{MountUtils} --- класс-обёртка над операциями монтирования/размонтирования;
            \item \texttt{UdevDeviceResolver} --- класс получения параметров устройств и монтирования;
            \item \texttt{UdevWatcher} --- класс, отслеживающий события с устройствами.
        \end{itemize}
        \item \texttt{essesnses/} --- содержит классы-сущности и их классы породители:
        \begin{itemize}[label=---]
            \item \texttt{Device} --- класс устройства;
            \item \texttt{DeviceBuilder} --- класс-создатель объекта устройства;
            \item \texttt{DeviceEvent} --- класс события подключения/отключения устройства;

        \end{itemize}
        \item \texttt{facade/} --- содержит класс \texttt{Facade}, унифицированный интерфейс доступа к подсистемам;

            \item \texttt{DevLogger} --- класс логгера;

            \item \texttt{FileException} --- класс исключения, связанного с файловой работой;
            \item \texttt{HttpServerError} --- класс исключения, связанного с \texttt{HTTP} сервером;

        \item \texttt{main.cpp} --- точка входа в программу.

\section{Развёртывание приложения}

Запуском серверной части управляет \texttt{systemd}~\cite{Systemd}. Файл конфигурации \texttt{systemd} располагается
по пути \url{/etc/systemd/system/usb-man.service}, его содержимое приведено в листинге~\ref{lst:systemd}. 

\includelisting
    {systemd}
    {Конфигурация \texttt{systemd}}

Исполняемый файл сервера находится по пути \url{/usr/local/bin/usbman}, файл базы данных лежит в 
\url{/var/lib/usbman/app.db}, скрипты для создания таблиц находятся по пути \url{/usr/share/usbman/sql/},
файл конфигурации для приложения находится в \url{/etc/usbman/config.txt}, его содержимое приведено 
в листинге~\ref{lst:config.txt}.

\includelisting
    {config.txt}
    {Конфигурация приложения}

Запуск клиентского приложения выполняется с помощью \texttt{systemd} и сервера \texttt{Nginx}~\cite{Nginx}.
Конфигурация \texttt{Nginx} располагается в файле \url{/etc/nginx/sites-available/usb-ui}, её содержимое приведено
в листинге~\ref{lst:usb-ui}.

\includelisting
    {usb-ui}
    {Конфигурация \texttt{Nginx}}

Файлы, отдаваемые \texttt{Nginx} лежат в папке \url{/var/www/usb-ui}. В ней находятся собранные
\texttt{*.html}, \texttt{*.js}, \texttt{*.css} и графические файлы иконок.

\section{Взаимодействие пользователя с приложением}

Пользователь может взаимодействовать с программным комплексом через \texttt{Web}-интерфейс, доступный
по адресу \url{http://localhost:5173/}. 

\subsection{Экран подключённых устройств}

По адресу \url{http://localhost:5173/list/} пользователю доступен список подключённых в данный
момент устройств. Над список есть 2 кнопки для перехода по экранам приложения. Слева находится кнопка с надписью 
<<Подключённые устройства>> (переход на описываемый экран), а справа от неё кнопка с надписью <<Доверенные 
устройства>> (переход в список доверенных устройств). Для каждого устройства отображаются следующие данные:

\begin{enumerate}
    \item путь монтирования;
    \item режим монтирования;
    \item название производителя;
    \item название устройства;
    \item серийный номер;
    \item идентификатор производителя;
    \item идентификатор устройства.
\end{enumerate}

Также справа от каждого устройства отображается иконка добавления в список доверенных устройств с изображением
знака \texttt{+}, при наведении указателя мыши на которую всплывает подсказка с текстом \texttt{<<Добавить>>}. В случае, 
если устройство уже доверено, справа отображается иконка с изображением знака \texttt{\checkmark}, при наведении курсора на которую 
всплывает подсказка с текстом \texttt{<<Уже добавлено>>}. При наведении указателя мыши на любую из кнопок курсор 
меняется на символ \includegraphics{inc/img/mouse.png}. Макет описанного экрана представлен на рисунке~\ref{img:interface_1}.

\subsection{Экран доверенных устройств}

По адресу \url{http://localhost:5173/whitelist/} пользователю доступен список доверенных устройств. Над списком аналогично 
находятся кнопки навигации. Для каждого устройства отображаются следующие параметры:

\begin{enumerate}
    \item название производителя;
    \item название устройства;
    \item серийный номер;
    \item идентификатор производителя;
    \item идентификатор устройства;
    \item дата истечения доверия.
\end{enumerate}

В случае если дата истечения доверия не задана(устройство доверено бессрочно), то в поле <<Доверено до>> отображается шаблон
даты \texttt{дд.мм.гггг}.

Также справа от каждого устройства отображается иконка удаления из списка доверенных устройств с изображением
символа \includegraphics[height=2.5ex]{inc/img/trash.png}, при наведении указателя мыши на которую всплывает подсказка 
с текстом \texttt{<<Удалить>>}.
